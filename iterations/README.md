# 迭代目录说明

本目录记录项目每次功能迭代的完整过程，帮助团队了解项目演进历程，为未来开发提供参考。

---

## 文件命名规则

```
ITERATION-<序号>-<简短描述>.md
```

- 序号为三位数字，从 `001` 开始，每次严格递增 1，**不可跳号**
- 简短描述使用中文，言简意赅

**示例**：
```
ITERATION-001-项目初始化与首页.md
ITERATION-002-图片压缩工具.md
ITERATION-003-首页性能优化.md
```

---

## 迭代文档模板

每个迭代文件必须包含以下结构（可根据实际情况增减字段）：

```markdown
# ITERATION-XXX：标题

**迭代时间**：YYYY-MM-DD
**迭代类型**：新功能 / 功能增强 / Bug 修复 / 性能优化 / 重构
**影响范围**：列出受影响的文件或模块

---

## 背景与目标
> 简述为什么要做这次迭代，解决什么问题或满足什么需求。

## 新增 / 修改内容
- 功能点 1
- 功能点 2

## 涉及文件

| 文件路径 | 操作类型 | 说明 |
|---------|---------|------|
| src/... | 新增     | ...  |
| src/... | 修改     | ...  |
| src/... | 删除     | ...  |

## 注意事项
> 上线注意事项、兼容性说明、已知问题等。

## 后续优化
> 本次未完成，留待后续迭代处理的事项。
```

---

## 使用规则

1. **每次迭代必须新建文件**，已有迭代文档不得修改（历史记录只增不改）
2. **文件命名必须唯一**，序号严格递增，不可跳号
3. **新增工具时**，必须同步：
   - 在 `docs/tools/` 下创建工具说明文档
   - 更新 `docs/README.md` 中的工具文档列表
   - 更新根目录 `README.md` 中的已有工具列表
4. **架构变更时**（新增依赖、调整目录结构、更换框架等），必须更新根目录 `README.md` 中的技术栈和项目结构章节
5. **Bug 修复**也需要记录，以便追溯问题来源
6. **完成迭代后**，同步更新本文件下方的迭代历史表格
